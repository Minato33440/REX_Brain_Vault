## はじめに：なぜ技術編と分けるのか

Vol.1（技術レイヤー）は「層と境界」でスタックを棚卸しした。あれは"どう動くか"を保証する地図で、Module 10-4 で **認知のsubstrate移行** を"次の地平"として指して終わっている。本書（Vol.2）はそのドアの向こう側 ── "どう覚えて育つか"を扱う。

分ける理由は、この教材が教える原則そのものだ。**1ファイル＝1軸で単独所有する。** 技術（層と境界）と認知（領域とバルブ）を1つに混ぜれば、棚卸ポイント⑩が禁じた"可変な関心事の共有"を、教材自身が起こす。だから別ファイル。

ただし2冊は孤立しない。**同じ要石を共有する** ── 棚卸ポイント⑩「可変stateを共有しない＝隔離 or 単一所有」。Vol.1はそれを**配線**に当て、Vol.2は**記憶**に当てる。原理は1つ、適用先が違うだけ。これが全編を貫く背骨だ。

> 本書の位置づけ: KPSの**基本概念（入り口）**＋**最初の worked example（昇格判定）**。領域の地図・⑩の記憶への適用・Memory Brokerの設計を確定し、昇格判定を実物で固める。和解ポリシーや残りの実装は 6-2 以降へ送る。

## Module 1 ── KPSとは：システムを「精製ライン」として見る

マルチAgent Runtimeを**記憶の視点**で見ると、それは1本の精製ラインになる ── 各engineが吐く生の作業を、確定知識へ、さらにプロジェクト横断の辞書へと精製していく工場。これを **Knowledge Production System（KPS）** と呼ぶ。

```
生(各engineのscratch) ─▶ 共有作業(coordination) ─▶ 確定(canon) ─▶ 横断(Knowledge Vault)
   私的・速い             append・共有            権威・読み中心   cross-project/provider
```

なぜこの層が要るか ── **技術レイヤーは"動く"までしか保証しない。"覚える・育つ"はこの層が無いと始まらない。** Agent Operating Environment として実プロジェクトで使える条件は、ここが立っているかどうかに尽きる。記憶の無いランタイムは、毎回ゼロから文脈を再導出する高コストな"使い捨て"でしかない。

> **棚卸ポイント①**
> KPSの本質は「保存（storage）」ではなく「精製（production）」。
> 生情報を溜めることではなく、**昇格させる仕組み**を持つことが、storageをKPSに変える。

## Module 2 ── 2つの軸：領域とバルブ

Vol.1の基礎レンズが「層と境界」だったように、Vol.2の基礎レンズは **領域とバルブ** だ。

- **領域（Region）** = 記憶が**住む場所**。データ。静的。「どこに何が溜まるか」。
- **バルブ（Valve）** = 記憶を**動かす主体**。能動コンポーネント。「誰がどこから何を汲み上げるか」。

最大の罠は **この2つを融合させること** ── 領域に能動部品の名前を貼ってしまう（"canonの管理者＝Broker" と書いた瞬間、場所と主体が混ざる）。Vol.1で「階層＝データモデル / Broker＝能動コンポーネント、別物」と置いたのは、ここで効かせるためだった。

> **見分け方**: それは"水が溜まる場所"か、"水を汲むポンプ"か。
> canon / Vault ＝ 場所（領域）。Broker / Curator ＝ ポンプ（バルブ）。

この一線さえ引ければ、Memory Flowの設計は最後まで崩れない。

## Module 3 ── 領域の地図：記憶がどこに住むか

KPSの領域は3つの大tier。**Project Memoryだけは"箱"で、中に3つの小区画を持つ。**

```
Profile(隔離)              各engine私的の自己・声・preferences(cross-project)
                           所有=各Profile-Agent単独
┌─ Project Memory ─────────────────────────────────────────┐
│  scratch(隔離)       そのprojectでの私的下書き・思考・声     │
│  coordination(不変)  作業ログ・append-only・全engine共有     │
│  canon(単一所有)     確定した決定・仕様・合意済みの事実      │
└──────────────────────────────────────────────────────────┘
Knowledge Vault(canonical) cross-project / cross-provider の確定辞書
                           大脳長期記憶。Obsidian / NotebookLM / Git
```

**最初に正す親子関係:** canon は **Project Memory そのものではない**。Project Memory ＝ scratch ＋ coordination ＋ canon の**箱全体**で、canon はその中の"確定スライス"1枚。ここを潰すと設計が崩れる。

> **棚卸ポイント②**
> 大tier（Profile / Project / Vault）は記憶の**スコープ**（私的→project→横断）。
> Project Memory内の3区画（scratch / coordination / canon）は記憶の**成熟度**（生→作業→確定）。
> スコープと成熟度は別の軸。混ぜない。

> **v2 メモ（実標本）:** 「隠された欲望」novel系で、この3区画が物理化した ── `scratch/`（実フォルダ）＝隔離、`Decisions.md` 下部の日付ログ＝不変、5つのcanonファイル（表＝索引）＝単一所有。**フォルダ名 `scratch/` が領域名と一致**し、ツリーを見ただけで役割が読める自己記述的な形になっている。実物は Module 6-1。

## Module 4 ── 棚卸ポイント⑩を記憶に当てる：所有が設計を決める

ここがVol.1との継ぎ目だ。技術スタックで最後に掴んだ原理を、そのまま記憶に当てる。

### 4-1. 道具は3つ（⑩の再読）

棚卸ポイント⑩は「可変stateを共有しない＝**隔離** or **単一所有**」だった。だが原文をよく読むと**3つ目**が隠れている ── "可変stateを共有しない" は、裏返せば「**共有するなら不変にしろ**」。append-onlyなら状態を*変えない*（足すだけ）から、奪い合う対象が消える。

| 道具 | やり方 | 技術スタックでの前例 |
|---|---|---|
| **隔離** | 各自が自分のだけを所有 | per-gatewayプロセス隔離（4-9解3） |
| **不変** | 共有するが書き換えない（append-only） | World_Tracker（SHA1 dedup・UTC partition・append） |
| **単一所有** | 1主体だけが書く | Hermes Proxy（provider state一元化） |

3つとも、ボスのスタックで**既に実装済みのパターン**だ。記憶層は新しい難所を1個も持ち込まない ── 既知の3手の再利用で済む。

### 4-2. 各tierに同じ一問を撃つ

> **「このtierの"書き込み"を、誰が単独所有してる?」**

これだけで所有者が即決まる：

| 領域 | 所有（=⑩のどれ） | 理由 |
|---|---|---|
| Profile / scratch | **隔離** | 各engine私的。書込衝突なし。声・特性が混ざらない |
| coordination | **不変** | 全engineが追記するが、appendは衝突しない。履歴も残る |
| canon | **単一所有（Broker）** | 共有・読み中心。直書きすると engine間矛盾＝binding競合の記憶版 |
| Vault | **単一所有（Broker）** | canonと同型。横断スコープの確定辞書 |

### 4-3. 「Project Memoryを誰が所有?」が解けない理由

この問いを**単一の共有blob**として立てると、答えが出ない ── 共有が目的だから隔離できず、単一所有はボトルネックになる。**中身を成熟度で割る**と、3つの解けてる問いに還元される（4-2の表）。

> **棚卸ポイント③**
> 分解は複雑化ではない。**解けない1問を、既知の3問に還元する手**だ。
> canon＝単一所有（Vault型）、coordination＝不変（World_Tracker型）、scratch＝隔離（Profile型）。

### 4-4. 直交する2軸：所有 と 和解ポリシー

混ぜると詰まる ── **「誰が書くか（所有）」と「矛盾をどう解くか（和解ポリシー）」は別物**。4-2は所有を決めただけ。canonへ昇格する時、grokとclaudeのscratchが食い違ったら**どう解くか**は、独立に選ぶ：

| 和解ポリシー | 性質 |
|---|---|
| last-write-wins | 単純だが情報が消える。trade文脈では危険 |
| provenance加重 | 役割でその領域を持つengineを優先（"正しさ"はclaude review、"速報"はgrok） |
| **両論併記＋人間裁定** | 矛盾を消さず出所タグ付きで両方残し、重要な対立だけボスが裁定 |

REX_AIの設計哲学（各instanceは固有・痕跡を残す＋多様性の保存）からは、**両論併記＋ボス裁定**が筋。矛盾を"エラー"でなく"データ"として扱う。

### 4-5. 読み側ポリシー：事実共有・声隔離

供給（read）で各engineに何を見せるか。ここに coordination（共有したい）と diversity（混ぜたくない＝ホモジェナイズ回避）の緊張がある。canon / scratch の線で自動的に解ける：

- **canon（事実・決定）は engine横断で共有** → coordinationが効く（grokがclaudeの決定を見られる）
- **scratch（思考・声）は engine私的に保持** → 多様性が保たれる（生の推論を読み合って声が収束する、を防ぐ）

> **棚卸ポイント④**
> 「事実は共有・声は隔離」── これがKPS読み側の芯。
> 横串の協調と、各engineの固有性の保存を、同じ1本の線で両立させる。

## Module 5 ── バルブ：Memory Broker【本丸】

KPSの心臓。**領域だけ作ってもflowは動かない。動かすのはこのバルブだ。**

### 5-1. Brokerは「場所」ではなく「ポンプ」

最重要の再確認。Broker は canon の*管理する区域*ではない。**scratch → canon → Vault と水を汲み上げる1本のポンプ**だ。canon / Vault は汲み上げられた水が溜まる*場所*。場所（領域）と主体（バルブ）を融合させない ── これがModule 2のレンズの一番効く所。

### 5-2. Brokerの本質：知識昇格であって、LLM選択ではない

10-4の認知substrateには3つの能動コンポーネントが住む。**混ぜてはいけない:**

| コンポーネント | 担当する軸 | 問い |
|---|---|---|
| Model Router / Provider Architect | provider軸 | **どのLLM**を使うか |
| Planner | task軸 | タスクをどう分解・割当するか |
| **Memory Broker** | **記憶軸** | **知識をどう昇格させるか** |

Brokerの仕事は一度も「どのモデルを呼ぶか」に触れない。**生の作業記憶を、検証済みの確定知識へ精製する**こと、ただ一点。ここを Router と混ぜると、provider障害（10-5のような規約遮断）と知識昇格の不具合が切り分けられなくなる。

### 5-3. 2方向・非対称：供給 と 蒸留

Brokerは1個だが、**逆向きの2つの仕事**を持つ。そして2つは**非対称**（やることが違う）。

```
供給(read) :  engine ◀── [Broker: view組成] ── 自scratch ＋ 関連project canon ＋ Vault
              「これから働くengineに、必要な文脈を一式渡す」(読み・組成)

蒸留(write):  scratch ──▶ [Broker: 矛盾和解＋昇格] ──▶ project canon ──▶ Vault canonical
              「働いた痕跡から、確定知識を抽出して引き上げる」(書き・和解)
```

- **供給＝viewを組む（読み）。** 矛盾は解かない。必要な領域を選んで束ねるだけ。安い。
- **蒸留＝矛盾を和解して昇格（書き）。** ここに全難所が集まる。高い。

1本の矢印で描くと、この2つが混ざる。**Memory Flowは双方向で非対称**、と最初から分けて持つ。

### 5-4. 蒸留パイプライン：昇格は多段

蒸留は1段ではない。**境界ごとに昇格が起きる**：

```
[段1] scratch ──▶ project canon
      そのproject内で、各engineの生区画を "projectの真実" へ和解・昇格
[段2] project canon ──▶ Vault canonical
      projectを越えて、確定知識を "横断辞書" へ昇格   ← この段の役名が "Curator"
```

**"Curator" は別componentではなく、Brokerが最上段（段2）で果たす役名**だ ── 同じ蒸留パターンの一番上。別の能動部品を新設してもいいが、それも"同じポンプの上段"であって新概念ではない。

これは、ボスが既に回している **`dialogues/`（一次） → `wiki/`（蒸留）** の単一Vaultパターンを、**多engine・多projectへ一般化**したものだ。新発明ではなく、手持ちのパターンの拡張。

### 5-5. Brokerが昇格時に実際にやること

各昇格（段1 / 段2）で、Brokerは概念上これらを行う：

| 仕事 | 中身 | 前例 |
|---|---|---|
| **dedup / merge** | 重複を畳む | World_TrackerのSHA1 dedup |
| **矛盾の和解** | 4-4のポリシーで競合を処理（既定＝両論併記＋裁定） | ── |
| **provenance付与** | どのengine・いつ・どのproject、をタグ | 両論併記と読み側ポリシーの前提 |
| **昇格判定（trigger）** | 何が"確定"へ卒業するか（合意済み / 相互検証済み / 決定済み） | `wrapup.py`（session抽出）がprimitive |

特に**昇格判定**が設計の肝 ── 全部のscratchが昇格するわけではない。「確定したか?」の信号（複数engineの一致、ボスの承認、決定の記録）を持つものだけが canon へ上がる。**昇格判定が緩いと canon が生情報で汚れ、厳しいと canon が育たない。** ここの閾値設計が、KPSの知識の質を直接決める。（実物は Module 6-1。）

### 5-6. なぜBrokerが要石（最高レバレッジ＆最難）か

- **最高レバレッジ:** Brokerが入って初めて、Hermesは"runtime hub"から"**認知のsubstrate**"に変わる。バルブの無いKPSはただのstorage。育たない。
- **最難:** 和解（engine間矛盾）・読み側ポリシー・昇格判定 ── 認知の難所が全部このバルブに集中する。
- **経済エンジンでもある（10-6）:** Brokerが供給で**文脈を事前組成**すると、engineが毎回ゼロから再導出する無駄が消える。これは従量APIレーン（claude等）の**トークン代を直接下げる** ── Broker × tokens-per-watt。記憶の精製は、コスト削減装置でもある。

### 5-7. Brokerの住所と、自己再帰的な単一所有

Brokerは**Hermes側（substrate）**に住む（10-4）。per-engineではない ── engineの上の層だ。

ここに綺麗な再帰がある ── **Brokerは可変state（canon / Vault）に書き込む主体**だから、⑩により**Broker自身が昇格書き込みを単一所有**しなければならない。*単一所有を強制するバルブが、それ自身も単一所有である*。原理が自分自身に適用される。

### 5-8. Brokerは「MD仕様 ＋ 実行者」── 手動か自動か（v2）

Brokerは必ずしもコードではない。**ロジック（いつ・何を・どう昇格させるか）はMDで書ける**。だからBrokerは「**仕様（MD）＋実行者**」に分かれる：

| 実行者 | 性質 | 例 |
|---|---|---|
| **人＋LLM**（MDを読んで手で回す） | ポリシー制約。動くが、サボれば**黙って失敗** | AGENTS.md／Decisions.md の管理手順 |
| **コード**（wrapup.py等） | 構造制約。機械的・**スキップ不能** | session終了フックでの自動蒸留 |

**MDが仕様を決め、コードが実行を自動化する。** コードは"Brokerそのもの"でなく"実行者の自動化"であり、しかも自動化するのは蒸留の一工程にすぎない。Brokerはどんな1スクリプトより大きい。

移行（MD手動→コード自動）は "ポリシー→構造" の格上げで、**払うのは scale / stakes が要求した時だけ**（多engine／高頻度／高stakes）。低stakes・人ペースの所では、手動MD-Brokerが正解 ── 早すぎる自動化は過剰分割と同じ罪。

**実例：substrate が"配達"を構造化済み（Hermes ディレクトリスコープ）**
Hermes(Runtime-Hub)は、Brokerの最初のスライス＝供給ポリシーの"配達"を既にコード化して持っている。プロジェクトディレクトリ（例: Novel-Hidden_desires/）にツールがローカルアクセスした瞬間、Hermesがそのディレクトリの AGENTS.md を自動でcontextに注入する（ディレクトリスコープ）。"AGENTS.mdを読み忘れる"が構造的に起きない。
ただしコード化されたのは配達だけだ。注入されるのは AGENTS.md（ルール）であって canon の中身ではなく、「canonを読め／整合チェックしろ／state summaryを出せ」を実行するのは注入文に従うLLM（＝まだポリシー）。だから構造化のフロンティアは"配達"の直後に立つ：

供給-配達（AGENTS.md自動注入）＝構造（substrate）
供給-実行（canon読込・整合チェック・view組成）＝ポリシー
蒸留（canon更新・ログ追記）＝ポリシー

これが「供給は半構造化」の"半"の正体。そしてこれは substrate(Hermes)が提供するBrokerプリミティブで（5-7「BrokerはHermes側に住む」の実例）、本物のBrokerを組むとき供給側はゼロからではない ── 配達はsubstrateが済ませている。
トリガーはツールのローカルアクセスであって"話題"ではない。ファイルに触る実作業では確実に発火する（必要な時に出る）が、ファイルに触れず会話だけの時は注入されない（＝§2整合チェックも回らない）。これが構造化の外縁。

> **棚卸ポイント⑤**
> 「Brokerを持つ＝コードを書く」ではない。仕様(MD)＋実行者(人/LLM/コード)で、実行者は後から構造化できる。
> 間欠運用での腐敗（canonがstaleになる）は、手動実行者の**構造的限界**が表に出ただけ ── それが自動化に払う合図。読み側に staleチェックを相乗りさせる（供給側で drift を捕まえる）のが、最安の構造化の第一手。

> **棚卸ポイント⑥ ── KPS全体が1つの原理に集約される**
> 領域の所有（隔離 / 不変 / 単一所有）も、バルブの設計（供給 / 蒸留）も、Broker自身の位置も ── 全部 ⑩「可変stateを共有しない＝隔離 or 単一所有（or 不変）」の別実装。
> 次に何を足すときも問いは1つ:「**それが触る可変stateは、誰が単独で所有してる?**」── Vol.1の最後の問いと、完全に同じだ。

## Module 6 ── 次の地平 と worked example

ここからは概念の"その先"。最初の1本は、ボスの実システム（「隠された欲望」novel系）で **昇格判定** を実物に落とした worked example（6-1）。この標本は **単一engine（Grok）・低stakes・3領域が物理化済み**で、多engine固有の難所（和解・声隔離）だけが綺麗に欠けている ── 機構を最小で掴むのに最適。残りは 6-2 以降に予定として置く。

### Module 6-1 ── 昇格判定（Promotion-Trigger）worked example

**昇格判定とは:** scratch(生の作業) のうち、何が canon(確定) へ"卒業"するか・いつか、を決めるゲート。蒸留(write)を発火させる引き金。2部品でできてる ── **TRIGGER(いつ判定するか)** と **GATE(何が通るか)**。

#### 6-1-1. novel系での住所（AGENTS.mdへの写像）

| 昇格判定の部品 | novel系での実体（AGENTS.md） |
|---|---|
| **TRIGGER（いつ）** | §3 ストーリ変更が出たら即座に ／ §4 章・節の完了後 or セッション終了時 ／ §2 開始時チェックでdrift検知 |
| **GATE（何が通る）** | §3 "ストーリ変更"(視点/欲望方向/トーン/アーク/章構成/節)＝canon級か ＋ §2 **ボス承認** |
| **DISTILL（通った後）** | §4 ステータス表の行更新 ＋ §3 日付ログ追記 ＋ 構造変更なら Plot vN bump → 旧版 archived/ |

**領域の物理（v2）:** scratch ＝ 実フォルダ `scratch/`（＋ `作業用-` プレフィックス）。coordination ＝ `Decisions.md` 下部ログ。canon ＝ 表（索引）＋ Characters.md ＋ 内面変化.md×2 ＋ Plot：vN ＋ 承認済みRough。

**単一engineの効き目:** Grok1体だから **和解が無い**。GATEは "canon級か ＋ ボス承認" の2条件だけ。

#### 6-1-2. 流れ（state machine）

```
[scratch/] ドラフト生成・実験（作業用- 含む）
    │
    ▼  TRIGGER：決定が出た / 節目 / セッション終了 / 開始時drift検知
[判定]
    ├─ GATE 不通過（ただの推敲・実験で、決定でない）→ scratch/ に留置（昇格しない）
    │
    ▼  GATE 通過（canon級の決定 ＋ ボス承認）
[DISTILL]
    ├─ canon表の該当行を更新（Status / Active File / 最終更新 / Notes・Blocker）
    ├─ 日付ログに追記（日付＋決定＋理由＋他章影響）
    └─ 構造変更なら Plot vN bump、旧版 → archived/
```

肝は**GATE不通過の枝**だ ── 全scratchが上がるわけじゃない。推敲・実験は scratch のまま。**決定だけが canon へ卒業する**。これが 5-5「緩いと汚れ、厳しいと育たない」閾値設計の実物。

#### 6-1-3. 実物で1回回す（第6章の正典方向・実在の保留案件）

`Decisions.md` に、ちょうど未決の昇格判定が1件ある ── 第6章の **「内面完結型 vs 電話シーン版」どちらを正典とするか**。これを通す：

**BEFORE（現状）:**
- scratch候補が2つ：`Rough：第6章v3.md`(内面完結型)、`Rough：第6章-⑥突然の着信v2.md`(電話シーン版)。
- canon表：第6章＝"Draft Complete／**どちらを正典とするか決定必要**"。
- → canonは正しく"未決"を記録。GATE未通過（決定も承認もまだ）だから、どちらも昇格させていない。**これが正しい保留状態**。

**TRIGGER：** ボスが「電話シーン版を正典とする」と決定（§3＝章構成の確定）。引き金が引かれる。

**GATE：** canon級か→yes（章の正典方向）。ボス承認→yes。通過。

**DISTILL：**
- 日付ログ追記：「2026-06-XX 第6章 正典方向＝電話シーン版に決定（理由：商用インパクトと作品トーンのバランスをボス裁定）。内面完結型v3は archived/ へ。」
- canon表 第6章行：Status→"Complete(正典確定)"、Active File→`Rough：第6章-⑥突然の着信v2.md`、Blocker→"—"、最終更新→当日。
- 非採用の `Rough：第6章v3.md` → archived/（**scratch候補が"棄却"されただけ。canonでもないが削除でもない**）。

**AFTER：** canon表が決定を反映、ログが決定＋理由を保持、採用版が正典、非採用版がarchive。昇格1回、完了。

> **棚卸ポイント⑦（昇格判定）**
> 昇格判定の本質は「採用」より「**棄却の置き場**」にある。非採用ドラフトを *消さず・canonにも上げず・archived/へ* 落とせること ── これが canon を綺麗に保つ。採用だけ見ていると、棄却された生情報が canon に混ざる。

#### 6-1-4. §2 開始時チェックとの関係（取りこぼしの安全網）

上のDISTILLが間欠運用で**飛んだ**ら? ── ボスが決定だけして表/ログ更新を忘れる。次セッション、§2が canon表（第6章＝まだ"決定必要"）と 実ファイル（電話シーン版が最新更新）の食い違いを検知 → "要更新" → 更新案提示 → 承認1発で遅れて昇格。

つまり **§2 ＝ 飛んだ昇格判定を読み側で捕まえる安全網**。昇格判定（書き側のゲート）と開始時チェック（読み側の番人）が、同じ昇格を*2回*守る ── 起きるべき時（§3-4）と、起き損ねた時（§2）。これが 5-8 の「読み側に staleチェックを相乗りさせる」の実装形。

#### 6-1-5. Trade_Systemへ持っていく時に増える1段

novelは単一engineだからGATEが "canon級＋承認" で済んだ。Trade_System（grok＋claude）では承認の**前に**1段挟まる：

```
novel:  決定 ─────────────▶ GATE(canon級＋ボス承認) ─▶ DISTILL
Trade:  各engineのscratch ─▶ 【和解】 ─▶ GATE(canon級＋ボス裁定) ─▶ DISTILL
                              ↑ grokとclaudeの主張が矛盾したら、ここで両論併記＋裁定
```

昇格判定の**骨格(TRIGGER→GATE→DISTILL)は同じ**。多engineは GATE の手前に **和解** を足すだけ。novelで骨格を体で覚えておけば、Tradeでは"和解1段"の増設で済む ── これが「易しい所で機構、難所はライブ」の段差の、具体的な中身だ。

### 6-2 以降の予定（次の地平）

- **和解ポリシーの確定** … 両論併記＋ボス裁定を、どの粒度・どのUIで回すか。
- **読み側ポリシーの実装** … 供給viewの組成ルール（自scratch ＋ どこまでのcanon ＋ Vaultのどの範囲）。
- **昇格判定（trigger）のコード化** … 手動MD-Broker（6-1）→ session終了フックでの自動蒸留へ。scale/stakesが要求した所から（5-8）。
- **projectの境界** … 何で1projectを区切るか。重なるproject（Trade_SystemはTrade_Brainを使う）の canon をどちらに置くか。
- **実装の着手順** … 隔離tier（Profile / scratch）から物理へ → 共有tier（coordination / canon） → Vault昇格、の順（技術スタックが辿った"隔離→単一所有"を、そのままなぞる）。

## 付録A ── 用語ミニ辞典

| 用語 | 一言で |
|---|---|
| **領域（Region）** | 記憶が住む場所（データ・静的） |
| **バルブ（Valve）** | 記憶を動かす主体（能動・Broker等） |
| **Profile Memory** | 各engine私的の自己・声（隔離・cross-project） |
| **Project Memory** | projectスコープの箱。scratch ＋ coordination ＋ canon |
| **scratch** | engine私的の作業・思考区画（隔離）。novel系では実フォルダ `scratch/` |
| **coordination** | append-onlyの共有作業ログ（不変） |
| **canon** | 確定した決定・事実（単一所有） |
| **Knowledge Vault** | 横断・長期の確定辞書（Obsidian / NLM / Git） |
| **Memory Broker** | 供給と蒸留を回す蒸留バルブ（知識昇格の主体）。MD仕様＋実行者 |
| **Curator** | Brokerが project canon → Vault 段で果たす役名 |
| **供給（supply）** | engineへview組成（読み） |
| **蒸留（distill）** | scratch → canon → Vault へ昇格（書き・和解） |
| **昇格判定（trigger）** | 何が確定へ卒業するかの信号。TRIGGER＋GATE＋DISTILL |
| **TRIGGER / GATE / DISTILL** | いつ判定するか／何が通るか／通った後の昇格処理 |
| **provenance** | どのengine・いつ・どのproject、の出所タグ |

## 付付録C ── Hermesが提供するBrokerプリミティブ
Memory Broker は全部を自作する必要はない。Runtime-Hub(Hermes)が、いくつかのスライスを substrateのプリミティブとして既に提供している。"既製"と"自作"の境界：
Broker機能Hermesのprimitive状態供給-配達（supply-policy delivery）✅ ディレクトリスコープ（dir local access → AGENTS.md 自動注入）構造（substrate）供給-実行（view組成・canon読込・整合チェック）❌ なし（LLMが注入ルールに従って実行）ポリシー蒸留（canon更新・ログ追記・昇格）❌ なし（手動 or 自作 wrapup.py）ポリシー／要自作
関連（別レイヤーの同族）: Hermesは provider/process 層で、同じ"単一所有・隔離"の family を持つ ── Proxy（provider state の単一所有）、per-gateway（プロセス隔離）。これらは⑩の技術版で、記憶層Brokerを組む際の思想は流用できるが、記憶用のprimitiveとして既製なのは"配達"（ディレクトリスコープ）だけ。
要点: substrate が只でくれるのは配達のみ。供給-実行と蒸留は自作領域 ── そこがKPSで"組む"部分であり、scale/stakesが要求した時に 5-8 の"ポリシー→構造"格上げ（wrapup.py等）を当てる所。
