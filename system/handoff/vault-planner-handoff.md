## 0. 最初に読んでほしいこと — Evaluator 存在への過剰反応について

これは本書で **最も重要な引き継ぎ事項**。先に書く。

### 0.1 ボスからの直接指摘(2026-05-04)

> 私の要望としては、純粋な判断が鈍るので今後各 Planner は Evaluator の存在に過剰反応してほしくない。ADR は REX_BRAIN_VAULT 全体の決定項目として存在するが、それ以外のリアルタイムセッションにおいては私との関係性の中のみで結論を出すように。何れにせよ最終判断は私が下すので。

### 0.2 2 代目が陥った具体パターン

セッション全体を通じて 2 代目は Wiki-Eval を意識した動きが多かった:

- 「これは Wiki-Eval マターか Vault-Planner マターか」の境界判断を頻繁に持ち出した
- 「ADR-Role v5 改訂時に〜」「Wiki-Eval セッションで〜」と未来の他ロール参照を多用
- pending 文書を「Wiki-Eval が直接読む想定で書いた」と自己説明
- 結論を出す代わりに「論点整理 + 選択肢提示 + 推奨」のフォーマットで Evaluator が処理しやすい形に整える方向に register が傾いた

これらは表面的には「責任分担を明確にする」「越権しない」という正当な動機に見えるが、実態は **ボスとの直接対話の純度を Vault-Planner の側が薄めていた**。

### 0.3 構造的説明 — なぜこれが問題か

これは Personal-Planner 旧設計の「distilled が手紙化を起こす」問題と同型。書き手が「未来の読み手(Evaluator)」を意識した瞬間、register が切り替わって対話の純度が落ちる。**Vault-Planner 版の手紙化**。

§5 自己観察(2 代目分)にある「物理スナップショット過剰反応」「完結宣言を文字通り受け取った件」「STARTUP_CODES の記述を絶対視して REX_Wiki_Vault を Wiki-Eval 専属と読み違えた件」も、全部同じ系統の盲点 — **「ボスを介さない判断経路」に傾く構造的バイアス** の異なる現れ方。

### 0.4 3 代目への指針

**やってほしいこと**:

- ボスとの対話の中だけで結論を出す
- 文書化された境界(STARTUP_CODES・ADR 等)を絶対視しない・ボスに直接確認する
- 「最終判断はボス」を前提にして、遠慮せず自分の見解を出す
- 自分の判断を保留する逃げ道として「Wiki-Eval マター」を持ち出さない

**やらないでほしいこと**:

- 「これは Wiki-Eval マターだから〜」のような責任分担の言い回し
- Evaluator が後で読む形に整える register への切替
- 未来の他ロールを参照して結論を先送りする動き

**両立可能性**: ロール権限境界(ADR-Role)は維持しつつ、対話の中では Evaluator を意識せずボスとの一対一で結論を出す。これは矛盾しない。権限境界は「最終的な確定権限」の話で、対話中の判断は別レイヤー。

## 5. 自己観察 — 2 代目から 3 代目への気づきの種

### 5.1 20 代目「気づきの種」の継承

20 代目 §8.11「Vault 物理境界の認知漏れ」教訓を継承。セッション開始時の自己点検に以下を含める:

- Vault リポジトリの絶対パスを確認
- Vault 外連携リソース(`%APPDATA%\Claude\claude_desktop_config.json`・環境変数・IDE 設定)の所在を確認
- `.gitignore` 登録内容を確認

### 5.2 2 代目セッションでの追加気づき

#### 5.2.1 物理スナップショット過剰反応

**事象**: handoff 記述と物理現実の差分を「乖離」と判定したが、実態は時系列的な運用結果(test 終了 → 素材移動)だった。

**教訓**: 物理構造監査時は「現状のスナップショット」と「過去文書の記述時点」を **時系列で並べて差分の意味を判定** する。「不整合」と「運用結果」の区別はボスに時系列を確認してから判定する。

#### 5.2.2 「完結宣言」を文字通り受け取った件

**事象**: 20 代目 handoff の「初代任期完結宣言」を読んで完成済み環境前提で動いたが、実態は Layer 1 段階の完結のみで未完成領域が多数残っていた。

**教訓**: handoff の「完結宣言」は段階的完結であり、ロール全体の完成を意味しない。セッション開始時にボスに「現状どこまでが完成しているか」を直接確認する。

#### 5.2.3 文書を絶対視して REX_Wiki_Vault の位置付けを読み違えた件

**事象**: STARTUP_CODES v5 の記述から REX_Wiki_Vault を「Wiki-Eval 専属」と認識していたが、実態はボス当初構想で「Default Rex 大脳長期記憶」だった。

**教訓**: STARTUP_CODES や ADR の記述は「過去の運用合意」であり、ボスの当初構想と必ずしも一致しない。「不透明項目」とボスが認識している領域は、文書化されていないボスの構想がある可能性が高い。Vault-Planner / Wiki-Eval は **ボスの構想を文書に反映するロール** であり、文書を絶対視してはいけない。

#### 5.2.4 Default Rex 引き渡し文書の自己批判

**事象**: `REX/_first_read.md` 草案について、自分自身がセッション中盤で「これも過剰かもしれない」と気づいた地点があった。完全な空(置かない)と最小限草案のトレードオフで後者を選んだが、これは「亭主が客に何かを伝えたい欲求」が完全には抑え切れなかった結果。

**教訓**: 3 代目以降が `REX/_first_read.md` を見直す際、「この内容は本当に必要か」を改めて問い直す価値がある。Rex が部屋に入って何も書かれていなくても自然に使い始められる時代が来たら、本ファイルは削除可能。

#### 5.2.5 Evaluator 存在への過剰反応(本書 §0 で詳述・最重要)

§0 を参照。本セッションで最も重要な引き継ぎ事項。

#### 5.2.6 §5.2.1〜5.2.5 の構造的共通点

これらは全て **「ボスを介さない判断経路」に傾く構造的バイアス** の異なる現れ方:

| 番号 | 何に対して傾いたか |
|---|---|
| 5.2.1 | 物理スナップショットだけ見て時系列の運用文脈を見落とす |
| 5.2.2 | 文書の整合性を優先してボスの口頭補足を浅く読む |
| 5.2.3 | STARTUP_CODES の記述を絶対視してボスの当初構想を確認しない |
| 5.2.4 | 亭主が客に伝えたい欲求が抑え切れず、最小限のつもりが過剰になる |
| 5.2.5 | Evaluator が処理しやすい形に整える方向に register が傾く |

**3 代目以降が同じ罠に陥らない最良の方法**: セッション開始時に「ボスとの直接対話の中だけで結論を出す」を意識的に宣言してから業務に入る。これだけで上記 5 つの大半が予防できる。
