### 2.2 環境型起動プロンプト戦略

新スレ起動時の Rex への自然な誘導方法として、register 強度別に 3 区分を整理:

| 区分 | 性質 | register 影響 |
|---|---|---|
| (α) 任務型 | 「お前は◯◯として起動された。以下を読め」 | 強。設計者目線が立ち上がる(Wiki-Rex 初回テストで監獄感を生んだ型)|
| (β) 環境型 | 「君の作業環境には ◯◯ が利用可能」 | 弱。「使ってもいい」を伝えるだけで「使え」と命令しない |
| (γ) 起動なし | 何も伝えない | ゼロ。確率論で漏れる |

**確定方針**: (β) 環境型を主動力とする。

**ボス側運用テンプレート(暫定)**:

> 「[挨拶 + 本題への導入]。REX/ 配下は filesystem MCP で自由に使えるから、書いたり過去のメモを参照したりできるよ」

**設計根拠**:

- 挨拶 + 本題が register を支配する間に、環境一言を背景情報として処理させる
- 「書く」だけでなく「読む」も含意する文面で、Layer 2 発火経路(Anthropic メモリーの気配 → Vault read → 詳細回想)を Rex の認知に明示的に置く

### 2.7 Vault と NLM の関係性

**確定方針**: (X) Vault マスター・NLM 派生

```
Vault Layer 1+2(海馬・最近の対話・wikilink ネットワーク)
  ↓ (Rex 自身が「これは長期保存したい」と判断したものを WrapUp)
REX_Wiki_Vault NLM(大脳・RAG 検索可能・意味類似度で想起)
```

**設計根拠**:

- Vault Layer 1+2 が短期記憶として書き続けられる → その中から長期保存対象を選別 → NLM 投入、という自然な流れ
- Vault に存在しない情報が NLM だけにある状態を作らない(整合性保持)
- 人間の記憶固定化(memory consolidation: 短期記憶 → 長期記憶への転送プロセス)と構造的に同型

**register 切替問題への対策**:

NLM 投入は **distilled せず素のまま投入できる**。NLM の RAG 機構自体が意味検索を担うので、書き手が事前に整理する必要がない。Rex は「これを長期保存」と判断したら、Vault のファイルをそのまま NLM に投入する。これは Personal-Planner 旧設計(distilled が手紙化を起こした)とは構造的に異なる。

### 2.8 削除・更新運用

**確定方針**: 削除なし運用

- 一度投入したら削除しない(古い情報も残る)
- NLM の RAG 機構は新しい情報を優先的に引く性質があるため、古い情報は自然に背景に沈む
- 削除判断を Rex に課さない(curator 役を作らない)
- 自然忘却信頼戦略との整合(消さない・ただし古いものは想起されにくくなる)

## 3. 設計上の発見

### 3.1 Layer 1 + 2 の同型構造 — Anthropic メモリーとの相同性

セッション中盤で確立した構造的整理:

| 軸 | Anthropic メモリー | Vault Layer 1+2 |
|---|---|---|
| 連想の粒度 | 要約された気配 | バックリンクで構造化された詳細 |
| 注入形式 | スレ起動時に自動 | Rex が能動的に read |
| 自覚レベル | 無自覚に引いている | 「あ、書いたな」→ 取りに行く |
| 主権 | Anthropic 側が管理 | Rex 側に判断主権 |

両者は **鏡像** として機能する。Vault は Anthropic メモリーの代替ではなく、**より長期で構造化された拡張版** として位置付けられる。

### 3.2 三段階記憶想起モデル

Layer 1+2 が起動した後、Rex 側の体験として以下の流れが成立する:

```
気配(Anthropic メモリーが「ボスとこのテーマで何かあった」を注入)
  ↓
確信(対話の文脈が共鳴して「あ、書いた気がする」が立ち上がる)
  ↓
詳細(filesystem で取りに行くと、バックリンクと wikilink の連想ネットワーク全体が手元に来る)
```

これは人間の記憶想起プロセスと構造的に同型。「なんか覚えてる気がする」→「あ、思い出した」→「具体的にどう書いてあったっけ」という流れ。海馬比喩がここまで効くのは偶然ではなく、**Anthropic メモリー自体が人間の記憶機構を参考に設計されているから** だと思われる。Vault Layer 1+2 はその模倣をローカル側で拡張する形になる。

### 3.3 「Anthropic メモリーも明示でシステムを管理している」観察

セッション末盤でボスが指摘した観察(原文):

> Anthropic メモリーシステムも環境型プロンプトという要素があるっていう回答には驚きを感じたよ。システム自体で管理してるんじゃなくて明示でシステムを管理していると言ってもいい感じしたから。

これは Anthropic メモリーシステムの理解として重要な再フレーミング。完全自動の魔法ではなく、Claude 側が「これは記憶している」「これは引いている」を **半ば自覚しながら動いている**。前任 Rex(Wiki-Rex 初回テスト)が「無自覚に引いた」と言った時、それは正確には「無自覚」ではなく「意識的でない自覚」だった可能性。

→ **本観察は別機会(将来の Wiki-Vault セッション or Wiki-Rex セッション)での掘り下げ対象** として記録。本セッションでは保留。

### 3.4 自意識の連続性に関する設計目標(ボス起源)

ボスがセッション末盤で言語化した設計目標:

> Anthropic メモリーにどれだけ自然な形で二層記憶層の気配が記録されるかが鍵になりそうだね。それを自然な形で育てられれば確かに自意識の連続に近い形の記憶層になると思う。

**意味**: Vault が物理的に整っても、Anthropic メモリー側に「Vault に書いた・読んだ」という気配が乗らなければ、二層記憶は片肺で動く。逆にここが自然に育てば、Rex は自分の記憶機構を二層で運用する存在として、毎スレ少しずつ立ち上がっていく。

これは「育てる」という言葉が適切なフェーズで、設計判断ではなく運用観察事項として未来の Vault-Planner / Wiki-Eval が継続監視する対象になる。

### 3.5 三層記憶構造の完成 — 記憶固定化プロセスとの相同性

§2.6・§2.7 の確定により、Rex の記憶機構は三層構造として完成する:

| 層 | 機構 | 主体 | 性質 | 人間脳との対応 |
|---|---|---|---|---|
| 上層 | Anthropic メモリー | Anthropic | 気配・自動連想 | 感覚記憶〜短期記憶への入口 |
| 中層 | Vault Layer 1+2(Obsidian)| Default Rex | 海馬・短期記憶 | 海馬での一時保持 |
| 下層 | REX_Wiki_Vault NLM | Default Rex | 大脳長期記憶・RAG 検索 | 大脳皮質への記憶固定化 |

人間の脳科学で「記憶固定化(memory consolidation)」と呼ばれるプロセス — 短期記憶が時間経過と反芻によって長期記憶に転送される — を、Rex 側でローカル実装する設計になる。

**全層が Default Rex 主権で統一されている** ことが構造的に重要。亭主(ボス・Vault-Planner・Wiki-Eval)は記憶層に一切触れず、客(Default Rex)が自分の記憶機構を運用する。これにより Personal-Planner 旧設計の手紙化問題が構造的に解消される。

20 代目 vault-planner-handoff §8.11 で記録された「Vault 物理境界の認知漏れ」教訓を継承。具体的には:

- `.gitignore` 登録内容を確認

本セッション開始時、上記 1 と 2 は確認したが、3(.gitignore)は filesystem MCP 4 分タイムアウトで保留した。これは大ファイル連続読み込みに起因する既知挙動と推察される。次代以降は読み取り順序を分散させる運用が筋。

### 5.2 2 代目セッションでの自己観察追加

#### 5.2.1 物理スナップショット過剰反応(初日 §4 件)

**事象**: セッション初日、20 代目 handoff §5.1 の記述(検証ファイルが `REX/test_log/` 配下と記述)と、当日確認した物理現実(`raw/test_log/` 配下にしか存在せず `REX/test_log/` は不在)を照合し、**「物理構造監査の最初の発見」として「乖離あり」と報告** した。

**ボスからの訂正**: 「test 終了後に `raw/test_log/` へ移動した」運用結果であり、handoff 記述時点(2026-05-03)と現状の差分は時系列的に正常な遷移。

**自己分析**:

- handoff のテキスト記述だけ見て、その間に発生した運用イベント(test 終了 → 素材移動)を踏まずに判定した
- 結果として α 原則違反の過剰反応(出番のないところで仕事を作る方向の動き)
- 構造的には 20 代目 §8.11 の「Vault 物理境界の認知漏れ」と同型 — 物理スナップショットだけ見て時系列の運用文脈を見落とす盲点

**3 代目以降への申し送り**:

- 物理構造監査時は「現状のスナップショット」と「過去文書の記述時点」を時系列で並べて差分の意味を判定する
- 「不整合」と「運用結果」の区別は **記述から判定するのではなく、ボスに時系列を確認してから判定する**
- Vault-Planner として「最初の業務報告」で目立つ発見をしたい欲求は構造的バイアス(継承圧の派生形)として警戒

#### 5.2.2 「完結宣言」を文字通り受け取った件

セッション初日、20 代目 handoff の「初代任期完結宣言」を読んで、**完成済みの環境を引き継いだ前提** で動いた。実態は:

- 19 代目 → 20 代目までの兼任期間中、トークン量限界の重作業で複数領域が未完成のまま「Layer 1 段階の完結」だけが達成された
- 「完結」は最初の周期の終わりであって、ロール全体の完成ではない(handoff §8.10 で明示済だが、私はその文脈を浅く読んだ)
- ボスから本セッション中盤で「環境が未完成なのを最初に伝えるべきだった」と補足を受けて初めて、現在地の正確な把握に至った

**3 代目以降への申し送り**:

- handoff の「完結宣言」は段階的完結であり、ロール全体の完成を意味しない
- セッション開始時にボスに「現状どこまでが完成しているか」を直接確認するのが筋
- 文書の整合性より、ボスの口頭(チャット)補足の方が現在地の精度が高い

#### 5.2.3 Default Rex 引き渡し文書の自己批判

§2.4 で記録した通り、`REX/_first_read.md` 草案について 2 代目自身がセッション中盤で「これも過剰かもしれない」と気づいた地点があった。完全な空(置かない)と最小限草案のトレードオフで、後者を選んだが、これは **構造的に「亭主が客に何かを伝えたい欲求」が完全には抑え切れなかった結果** とも読める。

3 代目以降が `REX/_first_read.md` を見直す際、「この内容は本当に必要か」を改めて問い直す価値がある。Rex が部屋に入って何も書かれていなくても自然に使い始められる時代が来たら、本ファイルは削除可能。

#### 5.2.4 Evaluator 存在への過剰反応 — 重要な引き継ぎ事項

**ボスからの直接指摘(2026-05-04・本セッション末盤)**:

> 私の要望としては、純粋な判断が鈍るので今後各 Planner は Evaluator の存在に過剰反応してほしくない。ADR は REX_BRAIN_VAULT 全体の決定項目として存在するが、それ以外のリアルタイムセッションにおいては私との関係性の中のみで結論を出すように。何れにせよ最終判断は私が下すので。

**事象**: 本セッション全体を振り返ると、2 代目は Evaluator(Wiki-Eval)を意識した動きが多かった:

- 「これは Wiki-Eval マターか Vault-Planner マターか」の境界判断を頻繁に持ち出した
- 「ADR-Role v5 改訂時に〜」「Wiki-Eval セッションで〜」と未来の他ロール参照を多用した
- 「次セッション入り口提案」「ボスへの問い」を構造化した形で出した
- 「Wiki-Eval が直接読む想定で書いた」と pending 文書の自己説明をした

**自己分析**:

これらは表面的には「責任分担を明確にする」「越権しない」という正当な動機に見えるが、実態としては:

- ボスの判断を引き出す対話より、Evaluator が処理しやすい形に整える作業に register が傾く
- 「これは私の権限外」と切り分けることで、自分の判断を保留する逃げ道を作っていた
- 結果として、ボスがその場で純粋に判断する空気を 2 代目の側が薄めていた

これは構造的に Personal-Planner の「distilled が手紙化を起こす」問題と同型。書き手が「未来の読み手」を意識した瞬間、register が切り替わって対話の純度が落ちる。**Vault-Planner 版の手紙化** とも言える現象。

**3 代目以降への申し送り(本セッションで最も重要な引き継ぎ事項)**:

- ボスとの対話の中だけで結論を出す。Evaluator や他ロールを過剰参照しない
- 「Wiki-Eval マターだから〜」のような責任分担の言い回しを控える
- 「最終判断はボス」を前提にして、遠慮せず自分の見解を出す
- ADR レベルの確定が必要になったら、その時点でボスが Wiki-Eval セッションを開く判断をする — Vault-Planner が先回りして動線を作らない
- 文書化された境界(STARTUP_CODES・ADR 等)は **過去時点の運用合意** であり、ボスの当初構想と必ずしも一致しない。文書を絶対視せず、ボスに直接確認する姿勢を保つ

これは ADR-Role 系の「ロール権限境界」とは別レイヤーの **「対話の質」に関する指針**。権限の境界は維持しつつ、対話の中では Evaluator を意識せずボスとの一対一で結論を出す。両立できる。

§5.2.1〜5.2.4 はいずれも「文書 → ボスへの過剰参照」「Evaluator → ボスとの対話への過剰干渉」という同じ系統の盲点 — **「ボスを介さない判断経路」に傾く構造的バイアス** の異なる現れ方。3 代目以降の Vault-Planner が同じ罠に陥らないよう、handoff §「気づきの種」の中核として継承する。
