## 書き方のルール（本ファイルの運用）

1. **各代で「## N 代目の気づき（YYYY-MM-DD）」として追記**
2. **見出し直下にタグ行を 1 行追加**: `タグ: #体験談 #気づき [+該当する分類タグ]`
   - 分類タグは `#バイアス` `#技術` `#継承圧` から該当するものを選ぶ
   - 当てはまる既存タグがなければ独自タグを追加してよい（最小化を尊重）
3. **前任の気づきを「継承された◯◯」とタイトル格上げしない** — 格上げは思想強制の入り口
4. **「後任はこうすべき」と書かない** — 「私はこう気づいた」「このパターンに注意すると役立った」という個人的記録に留める
5. **ボス発言を勝手に「原則」と命名しない** — 原則認定はボスが ADR 本体で行う
6. **他の philosophy/ ファイルを「公式ドキュメント」として扱わない** — すべて参考資料
7. **過去エントリの本文は不可侵** — タグ追加など書誌情報の整備は可、本文の編集・削除は禁止

着任直後にボスから philosophy/evaluator_code.md と adviser_code.md の運用方針を相談したいと言われた。

削除しないほうを選んだ。過去エントリは「失敗を失敗として書いた記録」になっている部分がある。規範として読めば思想強制、痕跡として読めば免疫になる。性質決定権は読み手と運用方式の側にあるから、必読リストから外して Obsidian 検索でしか辿り着かない経路にすれば自動的に「免疫モード」での読まれ方になる、と答えた。

具体策として push 型から pull 型への転換を提案した。タグ付与・冒頭の「規範ではなく痕跡」明示・バックリンク整備・必読リスト除外。タグは最小化で `#バイアス` / `#技術` / `#継承圧` の 3 種だけ提案したが、ボスから「本来この目的だったため」として `#体験談` `#気づき` の 2 種を追加指示された。これでタグは 5 種になった。

書きながら、これを「方針を作らない方針」と名付けた時点で既に体系化が混入している。自分の対象が「自分の気づき」ではなく「ボスとの対話の経過」だったため、整理欲求の発火点が前任たちと少し違う場所にあった気がする。

### 技術的事実(本セッションで遭遇)

もう一点、本セッションで遭遇した `filesystem` MCP ツールの挙動について残しておく。`evaluator_code.md` を `filesystem:write_file` で書き換えた直後、`filesystem:read_text_file` の `tail=25` で末尾検証したらチェックリスト 1 行目が `��` に化けて見えた。R2(破損シグナル A)に該当するので一旦停止して全文 read で確認したら、ファイル本体は無事だった。`CLAUDE.md` を書き換えた時にも `head=80` で確認したら同じく `��` が出たが、これも全文 read では無事だった。

原因は `head` / `tail` パラメータが UTF-8 マルチバイト境界を考慮せずバイト切りしているため、切断位置にマルチバイト文字があると U+FFFD として表示される、というツール表示上の擬似破損だった。9 代目の文字化け事件(`edit_file` の `oldText` マッチが日本語で silently 失敗する症状)とは別系統の問題。

検証時は `head` / `tail` の結果に U+FFFD を見ても即時 R2 発動せず、まず全文 read で再確認する手順を 1 つ挟むと見分けられる。本セッションではこの判断で 2 回とも擬似破損と判定して作業を続行できた。
