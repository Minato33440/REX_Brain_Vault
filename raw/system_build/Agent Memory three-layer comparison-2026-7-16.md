# AIエージェントの「記憶（Agent Memory）」三層比較まとめ

## 0. 大前提：なぜ「記憶」が必要なのか
AIエージェント（Claude CodeやCodexなど）は、セッションが変わると前の会話を全部忘れてしまうのが根本の問題。毎回同じ前提説明・同じファイル読み込みを繰り返すのは、小さく見えて毎日積み重なると大きな負担になる。
ポイントは「モデルの賢さ」と「記憶」を分けて考えること。どれだけ推論が優秀でも、過去の仕事が届かなければ判断材料がない。だから必要なのは「モデルの外側に、仕事の続きを残しておく設計」。
この動画では、その設計を担う3つのOSSを、担当する「層（レイヤー）」の違いで比較している。3つは競合製品ではなく、それぞれ違う高さ（層）の課題を解いているのがミソ。
記憶システムは「情報の流れ」で見ると分かりやすい：
入り口（出来事を受け取る）→ 保存・整理（背景ごと結び直す）→ 呼び戻し（作業に合う記憶を取り出す）→ 出口（実行・成果物へ渡す）。

---

## 1. rowboat（ロウボート）― 仕事をつなぐ「アプリ層」
一言でいうと：散らばった仕事を記憶と結びつけ、そのまま実行画面まで届けるAIコワーカー。

解決したい痛み：情報が「ない」ことより、情報が「つながっていない」こと。会議で決めた方針がメールやSlackの奥に埋もれ、結局また確認から始まる問題。
取り込み口：メール・会議記録・Slack・アシスタントとの会話など、バラバラの場所で生まれた仕事の断片をまとめて取り込む。
整理の仕組み：Obsidian風のグラフに入れ、バックリンクで関係づける。単発のメッセージではなく「仕事が動いた道筋」ごと思い出せる。
使い捨てにしない：前回の調査が今回の出発点になるよう、知識がグラフに積み上がっていく。
読む場所と使う場所が近い：メールクライアントを作業画面（ワークサーフェス）にして、読む・考える・調べる・コードを書くまでを同じ流れで扱える。
自動で動く：ユーザーが検索した時だけでなく、時刻トリガーや重要メール受信をきっかけに、バックグラウンドエージェントが過去の背景つきで処理を始められる。
実装：TypeScript製のOSS。

こんな人向け：仕事の断片が孤立して困っている人。「散らばった業務情報を探すところから、そのまま実行まで進めたい」人。

---

## 2.agenticow（エージェンティカウ）― 記憶を分岐する「インフラ層」
一言でいうと：記憶の「状態」をGitのように分岐・巻き戻しできる基盤。

キャッチフレーズ：「Git for Agent Memory」（エージェントの記憶にGitらしさを持ち込む）。作者は ruvnet、MITライセンス。
土台：まず一般的なベクトルメモリ（文章を数値化してベクターDBに保存し、意味の近い情報を検索する仕組み）がベース。
本題は「検索」ではなく「状態管理」：同じ記憶を持つエージェントに違う案を試させて結果を比べたい、という開発現場が想定シーン。
コピーオンライト方式：記憶を丸ごと複製せず、共通部分は親を指すポインターで共有し、新しく生まれた差分だけを重ねる。「大きな倉庫を建て直さず付箋だけ分ける」イメージ。

fork（フォーク）：同じ出発点から別の作業経路を作れる。一方の記憶がもう一方に混ざらない。
checkpoint（チェックポイント）：途中の記憶状態に目印を残せる。
rollback（ロールバック）：結果が悪ければ以前の地点に戻せる。失敗を「消して忘れる」のではなく「戻れる形で試す」。

操作コストが軽い：fork 1回が 約162バイト・0.47ms。checkpoint/rollbackも同じ。しかも元のメモリサイズに左右されず一定（中身ではなくポインター＋差分を扱うため）。
効いてくる使い方：並列実験（1つのベースから複数実験を混ぜずに走らせる）、ユーザーごとの専用メモリ（共通土台を配り、個別の変化だけをブランチで持たせる）。

こんな人向け：個人のメモを保存したい人ではなく、「記憶を使う仕組みを作る側（サービス基盤の開発者）」。ベクトルメモリに「分岐」と「復元」が必要かどうかが選ぶ基準。

## 3.claude-code-memory-setup ― 自分で育てる「個人セットアップ層（第二の脳）」
一言でいうと：Claude Codeの作業履歴を、自分の手元環境に引き継ぐDIYの仕組み。

位置づけ：完成したアプリを使うものではなく、自分の環境に組み込むOSS。作者は lucasrosati。
支える道具：

Obsidian の集約ボルト：全プロジェクトの壁を越えて「開発の記憶」を1か所に集める。
Graphiti（グラフィティ）：コードベースをナレッジグラフ化し、ファイルを片っ端から読まずに関係をたどれる道を作る。

残すもの：コードだけでは読み取れない\*\*「開発判断の背景（なぜこの設計にしたか）」\*\*。目的や状況から作業を再開しやすくなる。
効果：新しいセッションのたびに同じ範囲のファイルを読み直す負担を軽減。リポジトリの説明では 最大71.5倍少ないトークン。

⚠️ ただし「最大」であり全作業で同じだけ減る保証ではない。上限値として受け取るべき。

過去も取り込める：チャットインポートパイプラインで、以前の会話に埋もれた判断を第二の脳へ合流できる。
特徴：トルコ語（TR）のドキュメントもあり、英語以外でも読める点が公開OSSらしい。

こんな人向け：複数の案件をまたぐ人、Claude Codeの過去を自分の手で整えて「第二の脳」を育てたい人。ブランチ機能や基盤開発が欲しい人とは層が違う。

## 4.三層で選ぶ記憶戦略（比較の整理）
観点rowboat（アプリ層）agenticow（インフラ層）claude-code-memory-setup（個人層）担当する高さ仕事全体を囲む広い範囲ベクトルメモリの状態管理手元の開発習慣に最も近いDIY覚える対象人のやり取りに生まれた仕事情報と、その「関係」分岐できるベクトルメモリの「状態」開発者が後で再利用したい判断・進捗入り口新しい会話・メール・トリガー元のメモリ各プロジェクトの記録／過去チャット出口ワークサーフェス・外部ツール実行状態を各エージェントへ受け渡す中間地点Claude Codeへ背景と地図を戻す運用する人記憶を使うAIコワーカーを導入したい人基盤の開発者個人（自分で環境を育てる）
選び方の指針：似た項目に丸をつけて機能数で順位を決めると、本質（担当範囲の違い）を見失う。\*\*「自分が今いちばん繰り返しているコストはどこか」\*\*で選ぶのが正解。

連絡が分散し、決定事項を毎回探している → rowboat
同じ初期記憶から実験を派生させたい → agenticow
リポジトリを移るたびに説明し直している → claude-code-memory-setup

※ 3者を組み合わせる設計は考えられるが、公式に連携が示されているわけではない点に注意。

## 5.記憶が前提になる未来（まとめ）

rowboat … バラバラの仕事の断片を、背景までたどれる記憶にして、バックグラウンド行動まで運ぶ。
agenticow … 同じ土台から生まれる記憶を、利用者や実験ごとの状態として軽く分ける。
claude-code-memory-setup … 開発の続きを自分で引き継ぐための個人環境。

必要な記憶の形は「使う場所」で変わる。毎回ゼロから説明する働き方が変わるとき、Agent Memoryは追加機能ではなく「土台」になる——というのが動画の締めくくり。

必要なら、この内容をさらに短い要約版にしたり、特定の層（例：agenticowのコピーオンライトの仕組み）だけを深掘りしてまとめ直すこともできます。
