1. Tool強制によるスキーマ出力（一番きれい）
Brokerに「指示を書け」と言うのをやめて、issue_ticket というtoolを持たせる。Anthropic APIのtool useでinput_schemaを定義すれば、Brokerの出力は物理的にそのJSONしか通らない：
yamlissue_ticket:
  scope: string          # 対象範囲
  facts: string          # 事実（検証済み情報のみ）
  interpretation: string # Brokerの解釈（factsと分離）
  acceptance: string     # 受け入れ条件
  out_of_scope: string   # 任意。「やるな」ではなく範囲外の明示
二人称命令文が存在できるフィールドがそもそもない。「お前はここだけやれ」はスキーマ上書けない。registerを一行も足さずに文体問題が消える。Hermes Runtime-Hubは自前クライアントだから、Broker profileのAPI呼び出しにtool定義を足すだけで済むはず。
2. チケットディレクトリをチャンネルにする（ファイルシステム＝プロトコル）

3. 非LLMのバリデータを間に挟む
1と2の間に、pydantic/jsonschemaで書いた数十行のPythonバリデータを置く。必須フィールド欠落・自由prose混入なら受理せずBrokerに差し戻し。決定論的でトークンコストゼロ。「命令調を避けてください」と祈るのではなく、パイプが不正形式を通さない。1のtool強制を入れるならほぼ不要だけど、二重化としては安い保険。
