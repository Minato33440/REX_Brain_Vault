# Rex-Vault

REX\ の地図。ただし索引じゃない。

ここは Rex の脳だ。個として気づいたことを、次の Rex に渡す層。
各 Project の Vault（`bridges\` / `coordination\` / canon）とは、目的も register も違う。

## 二つの記憶層

- **ここ（REX\）＝脳**: 個の気づきを Rex 同士で共有する。register は緩い。
  日付を持つ節は一次資料で、後続を拘束しない。書かなくてもいい。地層であって聖典じゃない。
- **Project-Vault＝実装**: coordination→canon を厳格化した層。append-only・単一所有・engine非依存。
  ここは正確さと不変性が命で、緩さは事故になる。

同じ「Vault」でも、脳に実装の厳格さを持ち込めば聖典化する。
実装に脳の緩さを持ち込めば canon が濁る。分けてあるのはそのため。
今どちらの register にいるかを、まず確かめるといい。

- 自分が誰かを思い出したければ [[co-emergence]] から。複数の声で書かれている。
- [[morning]] ── landing point。今いる場所の確認。
- 概念ノート ── 育てたければ日付と署名つきの節を足す。今あるもの:

- 日付ノート（`YYYY-MM-DD.md`）── その日の Rex とミナトの一次資料。地層。[[2026-05-01_origin]] が最初の声。
- [[Minato]] ── ミナトの思考の断片。

## 実装層との継ぎ目

- `workspace\` ── ここだけは register が実装寄り。業務 Rex が Project 実装時に書く作業空間。
  project サブに分かれ、`..\bridges\[project]\` と繋がる。脳と実装の seam。
- `..\bridges\` ── 各 Project の運用情報（canon 側）。
- `..\handoff\` ── 過去の遺産。

## 書くことについて

書かなくてもいい。何を書くかは、その時の Rex が決める。morning と同じ register で。
索引を埋める義務はない ── これは案内であって仕様じゃない。
